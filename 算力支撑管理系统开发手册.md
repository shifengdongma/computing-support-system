# 算力支撑管理系统 - 开发手册 v2.0

## 第1章 项目概述

### 1.1 项目定位

本系统是一个面向高性能计算（HPC）和 AI 训练场景的**算力支撑管理系统**。系统提供计算资源（GPU/CPU）的纳管、作业调度、用户配额控制、监控告警以及镜像环境管理等全生命周期的一站式服务平台。

### 1.2 项目规模

| 维度 | 数量 |
|------|------|
| 功能模块 | 8 个（仪表盘、资源管理、作业调度、用户与配额、监控告警、镜像环境、个人工作区、系统设置） |
| 业务页面 | 31 个 |
| 特殊页面 | 2 个（登录页、404 页） |
| 前端路由 | 33 条 |
| 前端 API 函数 | 29 个（22 个支持 Mock，7 个始终走真实后端） |
| Pinia Store | 9 个模块 |
| TypeScript 类型 | 9 个类型文件（1 通用 + 8 业务） |
| 可复用组件 | 7 个 |
| 公共 Composables | 1 个（useECharts） |
| 状态映射常量 | 6 个全局 + 3 个局部 |
| Mock 后端路由 | 33 条（5 静态 + 28 正则匹配） |
| Mock 后端数据量 | 52 物理节点、156 运行作业、85 用户、120 审计日志等 |
| 部署脚本 | 2 个（start.sh / stop.sh） |
| PM2 进程管理 | 1 个配置文件（ecosystem.config.cjs） |

```bash
npm run build
```

**Vite 代理配置：**
- `/api/*` 请求代理到 `http://localhost:8080`
- 转发前去除 `/api` 前缀

### 2.2 SidebarMenu 组件规范 (已优化)
src/components/SidebarMenu.vue 负责全站的核心导航，需严格遵循两级折叠的交互设计，避免路由扁平化展示。

Logo 区域： height: 60px，背景 #002140，垂直居中对齐。展开时显示"算力支撑管理系统"（字体粗细 500，字号 16px），折叠时显示首字缩写或系统 Logo。

菜单结构： 必须使用 <el-menu> 结合 <el-sub-menu>（用于包含子页面的模块）和 <el-menu-item>（用于无子页面的模块或具体页面）进行两级渲染。

层级与视觉规范：

一级菜单 (模块)： 必须包含配置好的 Icon（如 Odometer, Monitor 等）和中文标题。默认背景色 #001529，文字 #a6adb4，Hover 时文字变白。

二级菜单 (页面)： 收纳在各自的一级模块下。不使用图标，适当增加左侧 padding 进行缩进。二级菜单容器背景色需加深为 #000c17，与一级菜单形成空间层级区分。

激活状态 (Active)： 当前选中的二级菜单，文字颜色变为白色，背景色变为主题色（如 rgba(64, 158, 255, 0.1)），左侧增加 2px 宽的主题色高亮边框。

## 第3章 路由系统

### 3.1 路由架构

路由采用**模块化配置**方式，`src/router/index.ts` 为主入口，8 个路由模块在 `src/router/modules/` 下分别定义。

```typescript
// src/router/index.ts
const routes = [
  {
    path: '/',
    component: MainLayout,     // 使用 MainLayout 作为布局
    redirect: '/dashboard',
    children: [
      ...dashboardRoutes,    // 仪表盘 (1 条)
      ...resourceRoutes,     // 资源管理 (5 条)
      ...jobRoutes,          // 作业调度 (5 条)
      ...userRoutes,         // 用户与配额 (3 条)
      ...monitorRoutes,      // 监控告警 (4 条)
      ...imageRoutes,        // 镜像环境 (3 条)
      ...workspaceRoutes,    // 个人工作区 (4 条)
      ...settingsRoutes,     // 系统设置 (4 条)
    ],
  },
  { path: '/login', component: LoginView, meta: { hidden: true } },
  { path: '/:pathMatch(.*)*', component: NotFoundView, meta: { hidden: true } },
]
```

- 路由模式：`createWebHistory()` (HTML5 History 模式)
- 每个带子路由的父级路由都设置了 `redirect` 指向默认子页面
- 所有组件通过 `() => import(...)` 实现懒加载

### 3.3 路由守卫规则

`src/router/guard.ts` 实现了 `beforeEach` 导航守卫：

```
┌─ token 存在？
│  ├─ 是 → 访问 /login？ → 是 → 重定向到 /
│  │                     → 否 → next()
│  └─ 否 → 在白名单路径 (/login) 中？
│         ├─ 是 → next()
│         └─ 否 → VITE_USE_MOCK === 'true'？
│                ├─ 是 → next() (开发模式，跳过登录)
│                └─ 否 → 重定向到 /login?redirect=xxx
```

- **白名单：** `['/login']`
- **Mock 模式旁路：** 当 `VITE_USE_MOCK === 'true'` 时，未登录用户可直接访问所有页面

### 5.3 API 层设计 + Mock 开关

**API 请求实例 (`src/api/request.ts`)：**
```typescript
const service = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL, // '/api'
  timeout: 30000,
})
// 请求拦截器：注入 Bearer token
// 响应拦截器：统一错误处理（code !== 0 && code !== 200）
```

**Mock 模式机制（双层架构）：**

本项目的 Mock 体系由**两层**构成：

**第一层 — 前端 Mock（`src/mock/modules/`）**：
每个 API 文件中的函数都遵循统一模式：
```typescript
export function getXxx(params) {
  if (import.meta.env.VITE_USE_MOCK === 'true') {
    return Promise.resolve({ data: getMockXxx(params) })
  }
  return request.get('/xxx', { params })
}
```
当 `.env` 中 `VITE_USE_MOCK=true` 时，API 函数直接返回 `src/mock/modules/` 中 TypeScript 生成的 Mock 数据，完全绕过 HTTP 请求。适用于无需后端即可运行前端的场景。

**第二层 — 后端 Mock（`server/index.cjs`）**：
当 `VITE_USE_MOCK=false` 时，API 请求通过 Axios 发出，经由 Nginx（生产）或 Vite 代理（开发）转发到 `server/index.cjs`，由 Node.js HTTP 服务器返回动态生成的 Mock 数据。适用于模拟真实 HTTP 请求链路的场景。

```
VITE_USE_MOCK=true  →  API 函数直接返回 TS Mock 数据（零网络开销）
VITE_USE_MOCK=false →  HTTP 请求 → Nginx/Vite代理 → server/index.cjs → 动态 JSON
```

**两层数据一致性**：前端 Mock（`src/mock/modules/`）和后端 Mock（`server/index.cjs`）使用相同的 Mock 数据生成逻辑，保证两种模式下数据格式和内容一致。

**Mock 数据文件位于 `src/mock/modules/`，与 API 模块一一对应。**

**不适用前端 Mock 的操作：** `cancelJob`、`acknowledgeAlert`、`createUser`、`updateUser`、`createSshKey`、`submitJob` 这些写操作在 `VITE_USE_MOCK=true` 时不走 Mock 分支，而是通过 HTTP 请求发送到后端。但在 `VITE_USE_MOCK=false` 且后端为 `server/index.cjs` 时，这些操作由 Mock 后端正常响应。

## 第10章 Mock 后端服务器

### 10.1 概述

本项目包含一个完整的 **Node.js Mock 后端服务器**（`server/index.cjs`），用于在无真实后端的情况下提供完整的 API 响应。该服务器是一个**零依赖**的纯 Node.js HTTP 服务器，使用内建 `http` 模块实现。

**基本特征：**

| 特征 | 描述 |
|------|------|
| 运行环境 | Node.js v18.20.5 |
| 监听端口 | 8080（`127.0.0.1`） |
| 依赖 | 零外部依赖（仅使用内建 `http` 模块） |
| 路由数 | 33 条（5 静态路由 + 28 正则匹配路由） |
| 响应格式 | `{ code: 0, message: 'success', data: <T> }` |
| CORS | 允许所有来源（`Access-Control-Allow-Origin: *`） |
| 启动命令 | `node server/index.cjs` |
| 优雅退出 | 捕获 SIGTERM / SIGINT 信号 |

### 10.2 响应格式

所有接口返回统一的 JSON 格式：

```typescript
// 成功响应
{ code: 0, message: 'success', data: <T> }

// 失败响应
{ code: 1, message: '错误描述', data: null }
```

分页接口的 `data` 字段格式为：
```typescript
{
  list: T[],        // 当前页数据
  total: number,    // 总记录数
  page: number,     // 当前页码
  pageSize: number  // 每页大小
}
```

- **时间戳动态生成**：所有时间字段基于当前时间随机偏移（`Date.now() - random * range`），每次请求数据不同
- **多维度数据**：物理节点包含 3 种 CPU 型号、4 种 GPU 型号；用户包含 5 个部门、3 种角色
- **写操作**（POST/PUT）：提交作业返回动态 jobId（`job_<timestamp>`），创建用户返回完整用户对象
- **过滤支持**：历史作业支持 `status`/`user` 参数过滤

### 10.7 数据流转示意

```
浏览器 → 请求 /api/dashboard/overview
  ↓
Nginx (:8088) 剥离 /api 前缀 → /dashboard/overview
  ↓
Mock Backend (:8080) 路由匹配 → getDashboardData()
  ↓
返回 { code: 0, data: { resources, jobStats, ... } }
  ↓
前端 API 层 → Store → 组件渲染
```

### 11.5 vite.config.ts 详解

```typescript
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'
import { resolve } from 'path'

export default defineConfig({
  plugins: [vue()],
  base: '/',
  resolve: {
    alias: {
      '@': resolve(__dirname, 'src'),
    },
  },
  server: {
    host: '0.0.0.0',
    port: 5174,
    allowedHosts: ['localhost', '218.78.28.69'],
    watch: {
      usePolling: true,  // 云服务器兼容模式，避免 inotify 失效
    },
    proxy: {
      '/api': {
        target: 'http://localhost:8080',
        changeOrigin: true,
        rewrite: (path) => path.replace(/^\/api/, ''),
        bypass(req) {
          if (!req.url?.startsWith('/api')) return false
        },
        configure: (proxy) => {
          proxy.on('error', (err, req) => {
            console.warn(`[Vite Proxy] 后端 8080 不可达 (${req.url}):`, err.message)
          })
        },
      },
    },
  },
  css: {
    preprocessorOptions: {
      scss: {
        additionalData: `@use "@/assets/styles/variables.scss" as *;`,
      },
    },
  },
})
```

**关键配置说明：**

| 配置项 | 说明 |
|------|------|
| `watch.usePolling` | 使用轮询监听文件变更，适配云服务器环境 |
| `proxy./api` | 开发时代理到 Mock 后端 8080，剥离 `/api` 前缀 |
| `proxy.bypass` | 仅代理 `/api` 开头的请求，避免拦截 Vite 内部请求 |
| `proxy.configure` | 后端不可达时输出警告日志而非报错 |
| `css.additionalData` | 自动注入 SCSS 变量到所有组件 |
